# Лабораторна робота  
## Проєктування DevOps-інфраструктури для кейсу

## Мета роботи

Метою лабораторної роботи є навчитися аналізувати програмний кейс з точки зору DevOps, визначати середовища роботи системи, описувати базову інфраструктуру, процес доставки змін, конфігурацію, моніторинг, логування та дії у випадку збою.

---

## Короткий опис системи

У межах лабораторної роботи розглядається вебзастосунок для керування завданнями команди, подібний до Jira. Система призначена для створення проєктів, додавання учасників команди, постановки завдань, відстеження їхнього статусу та контролю виконання роботи.

Система складається з трьох основних компонентів:

### Frontend

Frontend — це користувацький інтерфейс вебзастосунку. Через нього користувачі взаємодіють із системою: переглядають список проєктів, створюють завдання, змінюють їхній статус, авторизуються та працюють із командою.

Основні функції frontend:

- відображення сторінок застосунку;
- взаємодія з користувачем;
- надсилання запитів до backend через API;
- обробка відповідей сервера;
- відображення помилок та повідомлень.

### Backend

Backend — це серверна частина системи, яка містить бізнес-логіку та API. Він приймає запити від frontend, обробляє їх, взаємодіє з базою даних і повертає результат користувачу.

Основні функції backend:

- авторизація та автентифікація користувачів;
- обробка запитів frontend;
- керування користувачами, проєктами та завданнями;
- перевірка прав доступу;
- робота з базою даних;
- формування відповідей API.

### Database

Database — це база даних, у якій зберігається основна інформація системи.

У базі даних можуть зберігатися:

- користувачі;
- ролі користувачів;
- проєкти;
- завдання;
- статуси завдань;
- коментарі;
- історія змін.

База даних є важливою частиною системи, тому для production-середовища потрібно передбачити резервне копіювання, контроль доступу та відновлення після збою.

## Опис середовищ

Для правильної організації DevOps-процесу система повинна працювати в кількох середовищах.

### Development

Development — це середовище розробки. Його використовують програмісти під час написання та первинної перевірки коду.

Призначення development-середовища:

- локальна розробка frontend і backend;
- запуск системи на комп’ютері розробника;
- швидка перевірка змін;
- використання тестової або локальної бази даних;
- налагодження помилок.

У цьому середовищі допускаються часті зміни, експерименти та тимчасові помилки, оскільки воно не використовується реальними користувачами.

### Testing / Staging

Testing або staging — це тестове середовище, яке максимально наближене до production. Воно використовується для перевірки змін перед релізом.

Призначення testing/staging-середовища:

- автоматичне розгортання після проходження CI-перевірок;
- перевірка роботи frontend, backend і database разом;
- тестування нових функцій;
- перевірка виправлення помилок;
- проведення інтеграційного тестування;
- демонстрація змін перед випуском.

Staging допомагає знайти проблеми до того, як зміни потраплять до реальних користувачів.

### Production

Production — це основне робоче середовище, яким користуються реальні користувачі системи.

Призначення production-середовища:

- стабільна робота вебзастосунку;
- обслуговування реальних користувачів;
- зберігання реальних даних;
- контроль доступу;
- моніторинг стану системи;
- резервне копіювання бази даних;
- швидке реагування на збої.

У production не можна вносити неперевірені зміни. Усі оновлення повинні проходити через CI/CD-процес.
