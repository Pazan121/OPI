# Лабораторна робота

## Тема

Організація Software Configuration Management для навчального проєкту

## Мета роботи

Мета роботи — навчитися застосовувати базові практики Software Configuration Management у програмному проєкті: визначати configuration items, організовувати контроль змін, будувати traceability, фіксувати baseline і працювати з релізами.

## Короткі теоретичні відомості

Software Configuration Management — це підхід до керування змінами у програмному продукті. SCM охоплює не тільки вихідний код, а й документацію, конфігураційні файли, CI/CD-файли, релізи, теги та інші важливі елементи проєкту.

Git є інструментом контролю версій, а SCM є ширшим поняттям. SCM описує процеси керування змінами, фіксації стабільних станів, побудови traceability та підготовки релізів.

## Опис використаного проєкту

Для виконання лабораторної роботи було використано курсовий проєкт **«Кава-машина»**.

Проєкт реалізовано як вебзастосунок на Java з використанням Spring Boot, Maven та бази даних MySQL. Система дозволяє користувачу переглядати напої, обирати напій, перевіряти баланс і виконувати покупку. Адміністратор може керувати напоями, порціями та складниками.

Посилання на репозиторій:

```text
https://github.com/Pazan121/Coffee
```

## Завдання 1. Визначення configuration items

| № | Configuration item            | Призначення                       | Чому має бути під SCM                                             |
| - | ----------------------------- | --------------------------------- | ----------------------------------------------------------------- |
| 1 | `src/`                        | Містить вихідний код проєкту      | Будь-яка зміна коду може вплинути на функціональність програми    |
| 2 | `pom.xml`                     | Файл конфігурації Maven           | Визначає залежності, версію Java та процес збірки                 |
| 3 | `README.md`                   | Документація проєкту              | Містить опис проєкту та інструкції для запуску                    |
| 4 | `application.properties`      | Конфігураційний файл застосунку   | Впливає на підключення до бази даних і налаштування Spring Boot   |
| 5 | `.gitignore`                  | Файл правил ігнорування           | Запобігає потраплянню службових і тимчасових файлів у репозиторій |

## Завдання 2. Організація контрольованої зміни

Для контрольованої зміни було обрано оновлення документації проєкту. У файл `README.md` було додано інструкцію запуску курсового проєкту.

### Issue

Назва issue:

```text
Додати інструкцію запуску проєкту в README
```

Посилання на issue:

```text
https://github.com/Pazan121/Coffee/issues/1
```

### Branch

Для виконання зміни було створено окрему гілку:

```text
feature/update-readme-run-instructions
```

### Commit

Зміна була зафіксована commit:

```text
docs: add project run instructions
```

Commit hash:

```text
вписати commit hash
```

### Pull Request

Для включення зміни в основну гілку було створено pull request:

```text
docs: add project run instructions
```

Посилання на pull request:

```text
вписати посилання на PR
```

### Review

Під час перевірки було переглянуто зміни у файлі `README.md`. Було встановлено, що інструкція запуску є зрозумілою, зміни відповідають задачі, зайві файли до репозиторію не додані.

Після перевірки pull request було змерджено в основну гілку `main`.

## Завдання 4. Формування baseline

Після успішного merge було визначено стабільний стан системи та зафіксовано його як baseline.

Назва baseline:

```text
baseline-1.0.0
```

Дата створення:

```text
вписати дату
```

До baseline входять:

* вихідний код проєкту;
* файл `pom.xml`;
* документація `README.md`;
* конфігураційний файл `application.properties`;
* файл `.gitignore`;
* GitHub Actions workflow;
* історія змін у Git.

Цей baseline можна вважати стабільним, оскільки зміна була реалізована в окремій гілці, зафіксована commit, перевірена через pull request і після цього включена в основну гілку `main`.

## Висновки

У ході лабораторної роботи було розглянуто організацію Software Configuration Management для навчального проєкту **«Кава-машина»**. Було визначено основні configuration items, серед яких вихідний код, документація, конфігураційні файли, Maven-файл та GitHub Actions workflow.

Також було виконано контрольовану зміну через workflow `issue → branch → commit → pull request → merge`. Це дозволило простежити весь шлях зміни від її створення до включення в основну гілку.

Після merge було сформовано baseline `baseline-1.0.0`, який відображає стабільний стан проєкту. На його основі було створено навчальний реліз `v1.0.0` і підготовлено changelog.

SCM дозволяє краще контролювати зміни у проєкті, зберігати стабільні стани системи, швидше знаходити причину змін і безпечніше готувати релізи.
